# 나도 고양이 있어 명세

## 프로젝트명

나도 고양이 있어

## 목표

Windows 전용 데스크탑 펫 앱을 만든다. 사용자가 지정한 투명 창 영역 안에서 도트 고양이가 자유롭게 움직이고, 사용자의 마우스 조작에 말풍선이 아니라 행동으로 반응하게 한다.

## 핵심 컨셉

- "나만 고양이 없어" 밈에서 파생된 데스크탑 고양이 앱이다.
- 사용자의 화면 위에 작은 도트 고양이가 살고 있는 느낌을 제공한다.
- 고양이 외 기능은 넣지 않는다.
- 생산성 앱, 투두리스트, 게임 재화, 하우징, 전투 기능은 제외한다.
- 텍스트 대사보다 행동 모션 중심으로 반응한다.

## 기술 스택

- Electron
- HTML
- CSS
- JavaScript
- electron-builder

TypeScript는 사용하지 않는다. 초기 MVP 단계에서는 구조를 단순하게 유지하고, Codex가 빠르게 수정하기 쉬운 HTML/CSS/JS 기반으로 개발한다.

## MVP 기능

- 프레임 없는 투명 Electron 창을 제공한다.
- 창을 항상 위에 표시한다.
- 사용자가 창 크기를 조절할 수 있게 한다.
- 고양이는 사용자가 지정한 투명 창 영역 안에서만 이동한다.
- 고양이 없는 투명 영역은 마우스 클릭이 통과되어야 한다.
- 고양이 영역에서는 클릭, 드래그, 마우스 접근을 감지한다.
- 고양이는 랜덤 이동과 상태 전환을 수행한다.
- 마우스 접근과 드래그에 성격별 행동으로 반응한다.
- 커스터마이징 설정은 로컬에 저장하고 앱 재실행 시 불러온다.

## 2차 MVP 목표

1차 MVP의 "기술적으로 동작하는 고양이"를 넘어 실제로 살아있는 것처럼 느껴지는 도트 고양이 경험을 만든다. 핵심은 복잡한 게임 시스템 추가가 아니라 픽셀 퀄리티, 애니메이션 자연스러움, 행동 디테일, 상호작용 감성, 움직임 타이밍을 개선하는 것이다.

2차 MVP에서는 다음 축을 우선한다.

- 도트 에셋 리디자인과 작은 크기 식별성 개선
- 방향별 idle/walk 스프라이트와 품종별 silhouette 차이
- idle, walk, sleep, rub, startled, run 애니메이션 품질 개선
- 눈 깜빡임, 꼬리 흔들림, 귀 움직임, 호흡 같은 생동감 강화
- idle 행동 다양화와 랜덤 행동 테이블 정리
- 마우스 근처 시선, 장시간 방치 시 sleep 증가 같은 환경 반응
- 쓰다듬기 판정, 방향 감지, 성공 반응, 연속 반응 개선
- 성격별 행동 성향과 전용 반응 강화
- 그림자, 픽셀 렌더링, 설정 UI의 최소 시각 개선
- 장시간 실행과 다양한 창/해상도 환경 안정화

2차 MVP 완료 기준은 고양이가 실제 살아있는 느낌이 나고, idle 상태가 심심하지 않으며, 쓰다듬기 반응과 성격 차이가 체감되고, 픽셀 퀄리티와 움직임이 임시 수준을 벗어나는 것이다. 장시간 켜 두어도 거슬리지 않는 데스크탑 동반자 느낌을 목표로 한다.

## 3차 MVP 목표

3차 MVP는 현재의 "투명 창 안의 도트 캐릭터" 느낌에서 벗어나 Desktop Goose류의 오버레이 데스크탑 펫 경험으로 전환하는 것을 목표로 한다. 앱 창이나 설정 UI보다 화면 위에 작은 생명체가 존재한다는 감각을 우선한다.

핵심은 기능 확장이 아니라 다음 감각을 강화하는 것이다.

- 존재감
- 자연스러운 행동 리듬
- UI 제거
- 오버레이 몰입감
- 화면과의 일체감

3차 MVP의 구현 방향은 다음과 같다.

- 평상시에는 고양이 외 UI를 숨긴다.
- 창 경계, 회색 배경, 점선 outline, 리사이즈 핸들, 이름표를 기본 노출하지 않는다.
- 설정은 트레이 메뉴, 숨겨진 설정 모드, 최소 버튼 같은 별도 진입 방식으로 분리한다.
- Electron 투명 always-on-top 창은 화면 전체 오버레이처럼 동작하게 하고, 고양이 hit box 주변만 마우스 이벤트를 받게 한다.
- 행동 리듬은 더 느리고 덜 즉각적으로 조정한다. 짧게 움직이고 오래 멈추며, 반응 전 pause와 망설임을 둔다.
- 마우스 접근은 즉시 반응하지 않고 시선 처리, 짧은 멈칫, 성격별 거리 판단 후 행동하도록 만든다.
- Desktop Goose를 그대로 복제하지 않고, 고양이라는 주제에 맞는 조용한 오버레이 동반자 경험으로 재해석한다.

3차 MVP의 우선순위는 다음과 같다.

1. 창 영역감과 평상시 UI 제거
2. 설정 진입 방식 정리
3. 행동 빈도와 템포 조정
4. idle 감성과 화면 인식 느낌 강화
5. 마우스 상호작용 자연화
6. 렌더링 안정화와 최종 경험 점검

3차 MVP 완료 기준은 앱이 아니라 화면 위 생명체처럼 느껴지고, 창의 존재감이 거의 사라지며, 장시간 켜 두어도 산만하지 않고 Desktop Goose류의 오버레이 감성이 느껴지는 것이다.

## 4차 MVP 목표

4차 MVP는 "조용한 공존 기반 데스크탑 개입"을 목표로 한다. 3차 MVP가 화면 위 생명체처럼 느껴지는 오버레이 존재감이라면, 4차 MVP는 평소에는 조용히 같이 있다가 가끔 작은 흔적과 장난을 남기는 방향이다.

가장 중요한 원칙은 고양이가 계속 행동하거나, 계속 움직이거나, 계속 방해하지 않는 것이다. 조용히 같이 있는 시간, 아무것도 하지 않는 시간, 가끔 살아있는 것처럼 행동하는 순간이 기능 개수보다 중요하다.

4차 MVP의 구현 우선순위는 다음과 같다.

1. 존재감 리듬 시스템
2. 방해 강도 제어
3. 최소 소품 시스템
4. 약한 커서 관찰 행동
5. 성격별 조용한 개입 차이
6. 다중 모니터 구조 검토

기본값은 낮은 방해 강도로 둔다. 커서 경로 막기, 클릭 직전 도망, 드래그 방해, 지속적인 커서 추적 같은 실제 방해 행동은 기본 활성화하지 않고 옵션으로만 검토한다.

소품은 초기에 발자국, 털뭉치, 작은 공만 우선한다. 메모지처럼 텍스트 기반 오브젝트는 앱 느낌이 강해질 수 있으므로 후순위로 둔다. 소품은 최대 개수와 자동 제거 규칙을 반드시 가진다.

Rosebud 샘플 구현에서 재활용할 수 있는 아이디어는 게임 시스템이 아니라 내부 리듬 제어 요소로 제한한다. energy/socialTolerance 같은 숨은 상태값, sleep을 통한 energy 회복, 랜덤 blink 타이머, 정면/좌/우 이미지 에셋 매핑은 4차 MVP 후보로 둔다. hunger, food, toy, 말풍선, 하트 스파클처럼 게임성이나 텍스트 UI가 강한 요소는 기본 방향에서 제외한다.

다중 모니터 이동은 4차 MVP에서 실제 구현하지 않고 구조 검토와 리스크 기록 수준으로 제한한다. DPI 차이, 모니터 좌표계, 클릭 통과 경계 처리는 후속 MVP에서 실제 Windows 환경으로 검증한다.

4차 MVP 완료 기준은 고양이가 계속 행동하지 않고, idle 시간이 충분히 길며, 장난 행동이 드물고 작게 발생하고, 사용자가 작업 중이면 조용해지는 것이다. Desktop Goose류 감성은 기능보다 리듬으로 느껴져야 한다.

## 창 동작

- 대상 플랫폼은 Windows로 제한한다.
- Electron의 프레임 없는 투명 창을 사용한다.
- 창은 항상 위에 표시한다.
- 창 크기 조절을 허용한다.
- 고양이 이동 좌표는 창 내부 경계로 제한한다.
- 고양이가 없는 투명 영역은 클릭 통과가 가능해야 한다.
- 고양이 영역은 클릭, 드래그, 마우스 접근 감지 대상이다.
- 클릭 통과와 고양이 영역 이벤트 감지는 Electron의 플랫폼별 제약을 확인하면서 구현한다.

## 클릭 통과 제한사항

- 클릭 통과는 CSS 투명도나 `pointer-events`만으로 처리하지 않고 Electron 창의 `setIgnoreMouseEvents`를 사용한다.
- 고양이 없는 영역에서는 창 전체가 마우스 이벤트를 무시하므로 렌더러가 해당 영역의 클릭, 드래그, 휠 입력을 직접 처리할 수 없다.
- 고양이 영역 이벤트를 받으려면 마우스 위치가 고양이 영역 안으로 들어왔을 때 `setIgnoreMouseEvents(false)`로 되돌려야 한다.
- 현재 구현은 렌더러 마우스 좌표와 고양이 DOM 경계 기준으로 클릭 통과 상태를 전환한다.
- Windows 투명 창과 클릭 통과 동작은 운영체제, Electron 버전, 그래픽 환경에 따라 차이가 날 수 있으므로 실제 Windows 환경에서 재검증해야 한다.

## 고양이 동작

다음 상태를 구현할 수 있는 구조를 만든다.

- idle
- walk
- sleep
- pet
- rub
- startled
- run

고양이는 창 내부에서 랜덤 이동한다. 마우스가 가까워지면 성격에 따라 다르게 반응한다. 고양이 위에서 마우스를 드래그하면 쓰다듬기로 판정한다. 쓰다듬기 성공 시 말풍선 대신 행동 모션으로 반응한다.

예시 반응은 다음과 같다.

- 마우스에 부비적거리기
- 눈 감고 가만히 있기
- 한 걸음 피하기
- 놀라서 도망가기
- 마우스를 장난감처럼 따라가기

## 성격 프리셋

### 애교쟁이

- 마우스에 자주 다가간다.
- 쓰다듬기 반응이 길다.
- rub 모션 확률이 높다.
- 사용자 근처에서 오래 머문다.

### 새침이

- 마우스 접근 시 가끔 피한다.
- 쓰다듬기 허용 시간이 짧다.
- 과한 접촉 시 자리를 피한다.
- 반응 빈도가 낮지만 무시하지는 않는다.

### 겁쟁이

- 빠른 마우스 움직임에 놀란다.
- 갑작스러운 접근 시 도망간다.
- 천천히 접근했을 때만 쓰다듬기 반응이 좋다.
- 구석 쪽으로 이동하는 경향이 있다.

### 느긋이

- 이동 빈도가 낮다.
- idle/sleep 비중이 높다.
- 쓰다듬으면 가만히 눈을 감는 반응을 한다.
- 마우스 반응 속도가 느리다.

### 장난꾸러기

- 마우스를 자주 따라간다.
- 빠르게 움직이는 마우스를 쫓는다.
- 쓰다듬기 중 장난 모션으로 전환될 수 있다.
- idle 상태에서도 짧게 뛰거나 방향을 바꾼다.

## 커스터마이징

설정 가능한 항목은 다음과 같다.

- 고양이 이름
- 품종
- 눈 색상
- 털 색상
- 성격

앱 실행 직후에는 고양이를 바로 보여주지 않고 시작 설정 화면을 먼저 보여준다. 사용자가 설정을 완료하면 해당 설정이 적용된 고양이가 투명 창 안에 나타난다. 설정은 이후에도 다시 열 수 있어야 한다.

## 저장 방식

- 모든 설정은 로컬에 저장한다.
- 앱 재실행 시 이전 설정을 불러온다.
- 설정이 없거나 손상된 경우 기본값을 사용한다.
- 초기 구현에서는 단순한 JSON 기반 저장 방식을 우선 검토한다.

## 에셋 정책

- 초기 버전에서는 임시 도트 고양이 또는 단순 스프라이트를 사용한다.
- 처음부터 고품질 에셋 제작에 집중하지 않는다.
- 애니메이션 상태와 이미지 에셋 경로는 분리해서 관리한다.
- 나중에 스프라이트 이미지를 쉽게 교체할 수 있어야 한다.

## 현재 임시 스프라이트 구조 분석

현재 고양이는 실제 이미지 파일이나 sprite sheet가 아니라 HTML 요소와 CSS 도형으로 구성된 임시 스프라이트다.

- 루트 요소는 `index.html`의 `.cat`이며 위치, 상태, 방향, 품종, 색상, 성격은 `data-*` 속성으로 표현한다.
- 시각 파츠는 `.cat__sprite` 안의 `.cat__ear`, `.cat__face`, `.cat__eye`, `.cat__nose`, `.cat__tail`, `.cat__paw` 요소로 나뉜다.
- 기본 크기는 `.cat`과 `.cat__sprite` 기준 72x48px이며, 귀와 꼬리, 발은 몸통 밖으로 일부 돌출된다.
- 방향 전환은 별도 좌우 이미지가 아니라 `.cat[data-cat-direction="left"] .cat__sprite`의 `scaleX(-1)` 반전으로 처리한다.
- 상태 애니메이션은 `.cat[data-cat-state="idle"]`, `walk`, `rub`, `sleep`, `startled`, `run` 선택자와 CSS `@keyframes`로 처리한다.
- 품종 차이는 현재 `fold`의 귀 높이/기울기, `bobtail`의 꼬리 길이만 CSS override로 표현한다.
- 털, 귀, 눈, 그림자 색상은 `--cat-fur`, `--cat-ear`, `--cat-eye`, `--cat-shadow` CSS 변수로 주입한다.

현재 구조의 장점은 이미지 에셋 없이 빠르게 상태와 커스터마이징을 검증할 수 있다는 점이다. 단점은 픽셀 아트의 실루엣과 프레임 품질을 세밀하게 제어하기 어렵고, 방향별/상태별 고유 프레임을 만들수록 CSS 파츠 조합이 복잡해진다는 점이다.

2차 MVP의 실제 도트 에셋으로 넘어갈 때는 다음 구조를 우선 검토한다.

- `.cat` 루트와 `data-cat-state`, `data-cat-direction`, `data-cat-breed`, 색상/성격 데이터는 유지한다.
- `.cat__sprite` 내부 파츠는 sprite sheet 또는 상태별 이미지 레이어로 교체 가능하게 한다.
- 방향별 프레임은 `scaleX` 반전에만 의존하지 않고 필요한 경우 좌/우 전용 프레임을 둔다.
- CSS 변수 기반 색상 커스터마이징을 유지할지, 품종/색상별 에셋 세트를 분리할지는 실제 베이스 디자인 제작 후 결정한다.

## 스프라이트 교체 가능 구조 확인

현재 렌더러 로직은 고양이 시각 파츠를 직접 조작하지 않고 `.cat` 루트의 공개 상태값만 변경한다. 따라서 다음 속성을 유지하면 HTML/CSS 파츠형 임시 스프라이트를 이미지 기반 스프라이트로 교체할 수 있다.

- 상태 계약: `.cat[data-cat-state]`는 `idle`, `walk`, `sleep`, `rub`, `startled`, `run` 값을 가진다.
- 방향 계약: `.cat[data-cat-direction]`은 `left`, `right` 값을 가진다.
- 품종 계약: `.cat[data-cat-breed]`은 현재 `domestic`, `fold`, `bobtail` 값을 가진다.
- 색상 계약: `.cat`에는 `--cat-fur`, `--cat-ear`, `--cat-eye`, `--cat-shadow` CSS 변수가 주입된다.
- 위치 계약: `.cat`의 `left`, `top`, `width`, `height`, `transform: translate(-50%, -50%)` 기준으로 이동 경계와 마우스 hit box를 계산한다.

교체 시 유지할 최소 DOM 경계는 다음과 같다.

- `.cat` 루트는 유지한다.
- `.cat__name`은 이름 표시와 무관한 독립 UI로 유지한다.
- `.cat__sprite`는 고양이 본체의 유일한 시각 컨테이너로 유지한다.
- `.cat__sprite` 내부의 현재 파츠 요소는 실제 sprite sheet 전환 시 제거하거나 숨길 수 있다.

교체 전 확인해야 할 리스크는 다음과 같다.

- 현재 상태 애니메이션은 `.cat__eye`, `.cat__tail`, `.cat__paw` 같은 내부 파츠 선택자에 의존한다.
- 이미지 기반으로 바꿀 경우 상태별 `background-position`, `background-image`, `steps()` 프레임 수를 별도 테이블로 관리해야 한다.
- `.cat`의 72x48px 기준 hit box를 바꾸면 이동 경계, 클릭 통과 판정, 쓰다듬기 판정이 같이 영향을 받는다.
- 방향별 전용 이미지가 생기면 기존 `scaleX(-1)` 규칙은 fallback으로만 사용해야 한다.

결론적으로 현재 구조는 `.cat` 루트의 상태 계약을 유지하는 조건에서는 교체 가능하다. 다만 실제 sprite sheet로 넘어가기 전 `.cat__sprite` 내부 구현을 CSS 파츠 전용 선택자에서 이미지 프레임 전용 선택자로 분리하는 작업이 필요하다.

## 2차 MVP 베이스 도트 고양이 디자인

11번 리디자인 기준 베이스 고양이는 HTML/CSS 파츠형 도트 스프라이트로 먼저 구현한다. 실제 이미지 에셋 전환 전까지 다음 디자인 기준을 사용한다.

- 기본 크기는 88x56px이며, 기존 72x48px보다 몸통/얼굴/꼬리 식별성을 우선한다.
- 실제 표시 레이어는 4px 단위 `.cat__pixel` box-shadow 픽셀 스프라이트로 구성한다.
- 기존 `.cat__body`, `.cat__face`, `.cat__tail` 등 파츠형 도형은 교체 전 fallback 구조로 남기되 화면에는 표시하지 않는다.
- 외곽선, 털, 그림자, 가슴 털, 눈, 코를 별도 픽셀 레이어로 분리해 색상 커스터마이징을 유지한다.
- 얼굴은 큰 눈, 코, muzzle 암시, whisker 암시를 픽셀 레이어로 표현해 작은 화면에서도 앞쪽 방향과 표정을 구분할 수 있게 한다.
- 방향은 현재 `right` 기본 프레임과 `left` 반전 프레임으로 구현하며, 방향 변경 시 180ms turning 프레임을 사용한다.
- walk 상태는 오른쪽/왼쪽 방향별 sprite 흔들림 규칙을 따로 두고, 발과 꼬리 애니메이션은 기존 상태 계약을 유지한다.

품종 정의는 다음과 같다.

- `domestic`: 기본 코리안 숏헤어. 표준 몸통, 긴 꼬리, 선 귀를 사용한다.
- `fold`: 접힌 귀와 조금 낮고 둥근 체형을 사용한다.
- `bobtail`: 짧은 꼬리와 짧은 몸통을 사용한다.

작은 크기 식별성은 다음 기준으로 본다.

- 외곽선은 3~4px 도트 단위로 유지한다.
- 눈은 7x9px 기준으로 얼굴 안에서 가장 먼저 보이는 색상 포인트가 되게 한다.
- 귀, 꼬리, 발은 몸통과 겹치지 않는 독립 silhouette을 가진다.
- 색상 커스터마이징 후에도 가슴 털, 그림자, 귀 안쪽 색이 분리되어 보이게 한다.

우선순위는 다음과 같다.

1. Electron 창 동작 검증
2. 투명 창, 항상 위, 크기 조절 검증
3. 클릭 통과 검증
4. 고양이 이동 로직 구현
5. 마우스 접근 감지
6. 쓰다듬기 판정
7. 성격별 행동 차이
8. 커스터마이징 저장
9. 실제 도트 고양이 에셋 교체

## 개발 원칙

- 한 번에 크게 바꾸지 않고 최소 작업 단위로 나누어 수정한다.
- 기존 코드가 있다면 삭제하거나 갈아엎지 않고 필요한 부분만 최소 수정한다.
- 문서, 커밋 메시지, PR 제목, PR 본문, 코드 주석은 한국어로 작성한다.
- PR 제목에는 `[codex]` 접두사를 넣지 않고, PR 생성 후 가능하면 자동으로 머지까지 진행한다.
- 실제 기능 구현 전에는 명세와 작업 목록을 먼저 정리한다.
- Windows 전용 Electron 동작 검증을 우선한다.
- MVP 범위를 넘는 기능은 추가하지 않는다.
- 말풍선이나 대사보다 행동과 모션으로 반응을 표현한다.
- 컨텍스트 길이, 토큰 사용량, 파일 탐색 비용, 코드 중복을 줄이는 방향으로 구현한다.
- 기능별 파일 분리를 유지하되 새 파일 생성 전 기존 파일 재사용 가능성을 확인한다.
- 설정값, 상태명, 이벤트명, 타입명은 상수나 설정 파일로 관리한다.
- 반복 로직은 공통 함수, 공통 모듈, 공통 설정 데이터로 분리한다.
- 이벤트, 입력, 저장, 상태 전환, 애니메이션 전환은 전용 시스템이나 전용 모듈로 공통화한다.
- DOM 요소는 반복 조회하지 않고 캐싱해서 사용한다.
- 함수는 가능하면 50줄 이하로 유지하고 하나의 역할만 맡긴다.
- 초기 MVP에서는 불필요한 라이브러리를 추가하지 않는다.
