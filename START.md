# START.md

## 시작 지침

이 프로젝트는 Windows 전용 데스크탑 펫 앱인 "나도 고양이 있어"를 만든다. 초기 MVP에서는 Electron, HTML, CSS, JavaScript, electron-builder만 사용한다. TypeScript는 사용하지 않는다.

## 작업 순서

1. `spec.md`에서 MVP 범위와 제외 기능을 확인한다.
2. `TASKS.md`에서 가장 작은 미완료 작업 하나를 선택한다.
3. 작업 전에 현재 저장소 상태를 확인한다.
4. 기존 코드가 있다면 필요한 부분만 최소 수정한다.
5. 작업이 끝나면 `STATUS.md`에 진행 상황을 한국어로 기록한다.
6. 관련 변경만 커밋하고 한국어 커밋 메시지를 사용한다.
7. 작업 단위마다 한국어 제목과 본문으로 PR을 생성한다.
8. PR 제목에는 `[codex]` 접두사를 넣지 않는다.
9. PR 생성 후 같은 작업 단위 안에서 바로 머지까지 진행한다.
10. 머지가 불가능하면 `STATUS.md`에 이유를 한국어로 기록한다.

## 현재 우선순위

1. Electron 기본 구조 만들기
2. 투명 창, 항상 위, 크기 조절 검증
3. 클릭 통과 검증
4. 고양이 이동 로직 구현
5. 마우스 접근과 쓰다듬기 판정 구현

## 빌드 방법

- 개발 실행: `npm start`
- 현재 플랫폼용 unpacked 앱 생성: `npm run pack`
- Windows 설치 파일 생성: `npm run build:win`
- Linux/WSL에서 Windows unpacked 실행 파일만 생성: `npx electron-builder --win dir`

빌드 결과물은 `dist/`에 생성한다. Windows 실행 파일은 실제 Windows 환경에서 최종 검증한다.

## 개발 원칙

- MVP 범위를 넘는 기능은 추가하지 않는다.
- 고양이 외 기능은 넣지 않는다.
- 말풍선 대사 시스템은 만들지 않는다.
- 행동 모션 중심의 반응을 우선한다.
- 에셋은 초기에는 임시 도트 고양이 또는 단순 스프라이트로 충분하다.
- 애니메이션 상태와 에셋 경로는 분리해서 관리한다.

## 토큰 절약 및 구조 원칙

- 컨텍스트 길이, 토큰 사용량, 파일 탐색 비용, 코드 중복을 최소화한다.
- 현재 진행 작업은 `TASKS.md`만 우선 확인하고, 완료된 0~20번 작업은 필요할 때만 `TASKS_ARCHIVE.md`에서 확인한다.
- 새 파일을 만들기 전에 기존 파일 재사용이나 기존 시스템 확장으로 해결 가능한지 확인한다.
- 역할이 다르면 파일을 분리하되, 무분별한 파일 생성을 피한다.
- 기능 파일은 `src/components`, `src/systems`, `src/services`, `src/storage`, `src/config`, `src/utils`, `src/assets`, `src/styles` 구조를 우선 검토한다.
- 동일하거나 유사한 로직은 복사하지 않고 공통 함수, 공통 모듈, 공통 설정 데이터로 분리한다.
- 상태명, 이벤트명, 타입명, 설정값은 하드코딩하지 않고 상수나 설정 파일로 관리한다.
- 복잡한 조건문 체인보다 상태 기반 구조와 별도 상태 전환 시스템을 우선한다.
- DOM 요소는 반복 조회하지 않고 초기화 시 캐싱한다.
- 입력과 이벤트 등록은 중앙 시스템에서 관리한다.
- 저장 로직은 전용 모듈로 공통화하고 `localStorage` 같은 저장소를 여러 파일에서 직접 호출하지 않는다.
- 애니메이션과 UI 전환 로직은 공통 함수나 공통 시스템으로 관리한다.
- 함수 하나는 가능하면 50줄 이하로 유지하고 하나의 역할만 맡긴다.
- 반복 계산 로직은 `utils` 모듈로 분리한다.
- 초기 MVP에서는 불필요한 라이브러리를 추가하지 않는다.
- 라이브러리를 추가할 때는 추가 이유와 대체 검토 여부를 `STATUS.md`에 기록한다.
- 수정은 필요한 부분만 하고 파일 전체 재작성, 불필요한 포맷 변경, 관련 없는 변경을 피한다.
- 작업 설명, 상태 보고, 문서, 커밋 메시지, PR 제목, PR 본문은 무조건 한국어로 작성한다.
